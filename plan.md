# 设计审查与优化计划

## 审查结论摘要

当前项目已经具备一个可运行的最小 CLI 雏形，但产品承诺、模板协议和实际实现之间存在明显错位。核心问题不是代码量不够，而是 v1 的“系统边界”没有收紧，导致文档承诺了多项尚未落地的能力。

### P0：必须先修正的问题

1. 初始化会把示例记忆直接复制到真实 memory 目录，污染用户初始状态。
    - 结果：新安装即带有伪造的 user/project/feedback 记忆，影响检索可信度。
    - 受影响文件：`lib/init.js`、`templates/user/`、`templates/project/`、`templates/feedback/`

2. 检索协议要求按 tags 过滤，但索引里根本没有 tags。
    - 结果：`AGENT-INSTRUCTIONS.md` 里的检索流程在实现上不可执行。
    - 受影响文件：`templates/AGENT-INSTRUCTIONS.md`、`lib/rebuild-index.js`

3. README 和维护协议承诺“自动衰减 / 模式提炼 / 每 10 次会话自动维护 / 索引超长自动收缩”，但 CLI 目前没有闭环支撑。
    - 结果：文档表达的是“自演化系统”，代码实现的是“手动维护的静态 Markdown 工具”。
    - 受影响文件：`README.md`、`templates/MAINTENANCE.md`、`templates/AGENT-INSTRUCTIONS.md`、`lib/maintain.js`、`lib/rebuild-index.js`

### P1：应尽快收敛的问题

1. `ttl` 语义前后不一致。
    - 模板写的是“多久后可衰减”，实现却是“超过阈值直接归档”。

2. 衰减规则与文档不一致。
    - 文档表达的是每 30 天继续乘以 0.8，当前实现只是“只要 >30 天，就衰减一次”。

3. 系统缺少“读取记忆后更新 `last_accessed` / `access_count`”的实现入口。
    - 当前协议要求 Agent 自己改文件，但 CLI 没提供任何辅助命令或库接口。

4. `MEMORY.md ≤ 200 行` 只是告警，不是实际约束。
    - 这与 spec 的验收标准不一致。

### P2：后续可增强的问题

1. frontmatter 解析器过于脆弱，缺少 schema 校验。
2. `init` / `maintain` / `rebuild-index` 都把 `process.exit` 和文件系统副作用耦合在库函数里，不利于测试和复用。
3. 缺少 fixture 测试，导致文档与实现漂移时不容易被发现。

---

## 优化目标

本轮优化建议先把项目从“概念完整”收敛到“契约可信”。v1 应优先保证以下几点：

- 初始化结果干净、可信、无伪数据
- 文档承诺与 CLI 能力完全一致
- 索引规则真实可执行
- 维护逻辑可预测、可测试、可回放

---

## 改进路线图

### Phase 1：收紧产品边界与文档承诺

**目标：** 先决定 v1 到底支持什么，不支持什么。

**建议动作：**
1. 二选一：
    - 方案 A：保留“自演化”定位，但补齐自动维护闭环。
    - 方案 B：将 v1 明确降级为“可维护的 Markdown memory scaffold”，把自动演化改成 roadmap。
2. 清理 README、Spec、模板中的超前承诺，避免把未来设想写成当前能力。
3. 给每个命令定义清晰的 contract：输入、输出、副作用、失败条件。

**产出物：**
- 更新后的 README
- 更新后的 spec
- 命令级行为说明

**验收标准：**
- README 中每一项能力都能在代码中找到对应实现
- 没有“Agent 应该自行完成，但工具链没有入口”的隐式要求

### Phase 2：修正初始化语义

**目标：** 让 `init` 生成的是可用系统，而不是带示例数据的 demo。

**建议动作：**
1. 不再把 `example-*.md` 复制到真实 memory 目录。
2. 示例文件改为文档样例，或移动到 `templates/examples/` 且默认不安装。
3. 真实初始化目录只保留：
    - `MEMORY.md`
    - `AGENT-INSTRUCTIONS.md`
    - `MAINTENANCE.md`
    - `TEMPLATE.md`
    - 空分类目录
4. `--force` 语义明确为“覆盖目标目录内容”，或改成更安全的“仅覆盖受管文件”。

**产出物：**
- 调整后的 `init` 逻辑
- 更清晰的模板目录结构

**验收标准：**
- 新生成的 memory 目录不包含虚构用户偏好或项目上下文
- 首次索引为 0 条记忆也能正常工作

### Phase 3：统一数据模型与检索契约

**目标：** 让索引、frontmatter、检索协议三者说同一种语言。

**建议动作：**
1. 在索引中补上 tags，或从协议中删除“按 tags 过滤”的要求，二者必须选一。
2. 明确 `ttl` 的单一语义：
    - 方案 A：TTL 到期即归档
    - 方案 B：TTL 到期才开始衰减
    - 方案 C：不用 TTL，只保留 `confidence` 与 `last_accessed`
3. 为 frontmatter 建立最小校验规则：
    - `type` 必须属于允许集合
    - `confidence` 必须在 0 到 1
    - 日期字段必须可解析
4. 定义文件命名规则与 slug 规则，避免 Agent 生成不可预测文件名。

**产出物：**
- 固化后的 frontmatter contract
- rebuild-index 的字段映射规范
- 简单校验器

**验收标准：**
- 模板、README、代码中的字段语义完全一致
- 非法 frontmatter 会被提示，而不是静默吞掉

### Phase 4：补齐维护闭环

**目标：** 让“维护”从口号变成真正可执行的流水线。

**建议动作：**
1. 明确区分两类维护：
    - `rebuild-index`：纯索引重建
    - `maintain`：衰减、归档、模式提炼、去重
2. 如果保留“每 10 次会话自动维护”，则至少新增一个可被 Agent 调用的轻量入口，例如：
    - `touch-memory <file>` 更新访问元数据
    - `session-end` 批量更新计数并在阈值触发 `maintain`
3. 将衰减改为基于时间片计算，而不是单次判断。
4. 将“索引 ≤ 200 行”从告警改为硬约束，并定义超限淘汰顺序。
5. 如果短期内不实现 pattern extraction / duplicate merge，就先从文档删除，不要保留半实现承诺。

**产出物：**
- 可闭环的维护命令集
- 明确的计数器更新机制
- 可复现的归档策略

**验收标准：**
- 从一次完整 session 到维护完成，有明确命令路径
- `maintenance_count` 不再只是文档字段，而是实际状态

### Phase 5：提高可测试性与可发布性

**目标：** 防止后续继续出现“文档说有，代码没有”。

**建议动作：**
1. 把纯逻辑与 CLI 输出/退出分离。
2. 为以下场景建立 fixture 测试：
    - 空目录初始化
    - 含 1 条、10 条、200+ 条记忆时重建索引
    - 30/60/90 天衰减
    - TTL 到期归档
    - 非法 frontmatter
3. 增加 `npm pack --dry-run` 校验，确认发布包只含必要文件。
4. 为 Windows 和 POSIX 路径行为各跑一轮测试。

**产出物：**
- 最小自动化测试集
- 发布前检查脚本

**验收标准：**
- 关键 contract 变更会触发测试失败
- 发布包结构稳定、可预测

---

## 推荐执行顺序

1. Phase 1：先裁剪承诺，避免继续扩大设计债务。
2. Phase 2：修 `init`，先保证安装结果可信。
3. Phase 3：统一索引与 frontmatter 语义。
4. Phase 4：决定是否真正实现“自演化闭环”。
5. Phase 5：补测试，锁定行为。

---

## 我建议的 v1 决策

如果目标是尽快发布一个可信版本，我建议采用下面的收敛策略：

1. v1 只承诺：`init`、`rebuild-index`、手动 `maintain`。
2. 暂时移除“每 10 次 session 自动维护”和“自动模式提炼”的对外表述。
3. 保留 `confidence decay + archive`，但把语义写准、实现补齐。
4. 先把系统做成“可靠的 Markdown memory scaffold”，再进入 v1.1 的自演化增强。

这样可以先把可信度做出来，而不是继续扩大概念范围。
